## 👥 Архитектурные решения

### Выбор LRU для кэша

**Почему LRU?**
- Простая в реализации
- Хорошая общая производительность
- Подходит для большинства паттернов доступа

**Альтернативы:**
- FIFO: дешевле, но хуже результаты
- Clock: лучше LRU, но сложнее
- LFU: отслеживает частоту, но сложнее

### Размер страницы 512 байт

**Почему 512?**
- Стандартный размер блока диска
- Минимизирует фрагментацию
- Баланс памяти и производительности

**Альтернативы:**
- 4 KB: меньше число операций I/O, но больше память
- 256 B: меньше память, но больше операций

### Мьютекс на API уровне

**Почему здесь?**
- Потокобезопасность из C#
- Предотвращение race conditions
- Гарантирует целостность данных

**Альтернатива:**
- Fine-grained locks: сложнее, можно в будущем

## 💾 Persistance

### Гарантии сохранения

1. **При записи:**
   - Данные в памяти (кэш)
   - Помечена как Dirty

2. При вытеснении из кэша:
   - Если Dirty: сохранить на диск
   - Обновить на диске

3. При Close:
   - FlushDirtyPages: сохранить все Dirty
   - Синхронизировать файловую систему
   - Закрыть файл

### Защита данных

- ✅ Все данные сохраняются на диск
- ✅ Sync при вытеснении страниц
- ✅ Flush при Close
- ✅ Bitmap отслеживает валидность
- ✅ Header содержит метаинформацию

### Типичные проблемы

| Проблема | Решение |
|----------|---------|
| Error -4 при Create | Удалить файл перед созданием |
| Error -1 при Open | Сначала создать (VMCreate) |
| Error -3 при Read | Проверить индекс (0 до size-1) |
| Медленная работа | Увеличить кэш (SetCacheSize) |
| DLL не найдена | Скопировать в PATH или рядом с .exe |
